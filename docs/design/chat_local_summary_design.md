# 本地会话摘要设计方案

## 1. 文档目标

本文档定义 AssistIM 后续“本地会话摘要”能力的需求边界、推荐实现方案和数据结构。

目标能力：

- 聊天摘要在后台静默处理
- 处理完成后通过 InfoBar 提示
- 摘要只保存在本地，且以加密形式持久化
- 摘要时间分段与聊天时间标签使用同一套规则
- 出现新的时间段时，触发一次增量摘要更新
- 删除会话时，一并删除本地摘要
- 图片、语音的摘要优先由模型自身完成
- 视频不纳入本期范围

本文档只输出设计，不包含本轮代码改动。

## 2. 当前基线与稳定约束

### 2.1 聊天时间标签规则

当前聊天列表中的时间标签并不是由 UI 单独维护的业务状态，而是由 `MessageModel` 在构建展示行时按相邻消息时间差插入 separator。

现有规则位于 `client/models/message_model.py`：

- 跨天时，开始新的时间段
- 同一天内，相邻两条消息间隔大于等于 5 分钟时，开始新的时间段

这意味着后续摘要分桶必须复用这套“时间分段规则”，而不能依赖“某个时间标签控件是否已经被创建”。后者会受到懒加载、滚动和分页影响，不是稳定业务事件。

### 2.2 本地存储加密边界

当前客户端本地数据库已经具备加密状态检测和 SQLCipher 兼容路径，相关入口位于 `client/storage/database.py`。

摘要设计应遵守以下约束：

- 摘要数据不落独立明文文件
- 摘要数据优先落本地数据库
- 摘要正文和媒体摘要字段在写库前再做一层应用侧密文封装
- 当数据库本身启用 SQLCipher 时，摘要获得数据库层和字段层双重保护

### 2.3 会话删除路径

当前本地删除会话走 `SessionManager.remove_session()`，最终会调用 `Database.delete_session(session_id)` 删除本地会话和消息。

后续摘要清理应挂在同一删除链路上，避免出现“会话已删但摘要残留”的状态分裂。

## 3. 需求归纳

## 3.1 功能需求

1. 每个会话按聊天时间标签同口径进行分桶摘要
2. 当前正在进行的时间段支持滚动摘要
3. 新时间段出现时，上一时间段固化为稳定摘要
4. 图片和语音消息进入摘要链路
5. 推荐回复后续可读取摘要结果，而不是临时重扫长历史
6. 摘要完成后给用户可感知但不过度打扰的提示

## 3.2 存储需求

1. 摘要仅保存在本地
2. 摘要持久化必须加密
3. 会话删除时同步删除摘要
4. 清空本地聊天状态时同步清空摘要

## 3.3 性能需求

1. 摘要生成不阻塞主聊天界面
2. 推荐回复阶段不应临时触发重型摘要计算
3. 图片、语音的理解结果应缓存复用
4. 同一时间段内重复更新应支持 debounce / 合并

## 4. 明确不做

- 不做视频摘要
- 不做基于 UI 控件创建事件的摘要触发
- 不做“每次推荐回复都全量扫描一周历史”
- 不做把 E2EE 明文发送到未授权远端服务

## 5. 推荐总方案

推荐采用“时间桶摘要 + 媒体摘要缓存 + 后台增量刷新”架构。

核心思路：

1. 复用聊天时间标签规则，把会话切成稳定时间桶
2. 每个时间桶维护一份文本摘要
3. 图片、语音先单独生成媒体摘要并缓存到本地
4. 时间桶摘要在生成时只读取文本消息和本地媒体摘要
5. 推荐回复后续只读取“最近消息 + 开放桶摘要 + 已关闭桶摘要 + 媒体摘要缓存”

这样可以把“长历史理解”从用户触发时刻前移到后台，减少推荐回复时延。

## 6. 时间桶模型

## 6.1 分桶规则

后续摘要系统必须复用当前聊天时间标签规则：

- `next.date() != current.date()` 时，开启新桶
- `abs(next - current) >= 300 秒` 时，开启新桶

建议后续把这套规则抽取为共享的时间分桶 helper，供：

- `MessageModel`
- 本地摘要任务
- 推荐回复上下文拼装

三处共用，避免 UI 展示规则和 AI 上下文规则漂移。

## 6.2 桶类型

建议区分两类桶：

- `open bucket`：当前仍可能继续追加消息的开放桶
- `closed bucket`：已被后续时间桶截断，不再追加的稳定桶

推荐行为：

- 同一时间桶内来了新消息，只刷新开放桶摘要
- 新时间桶出现时，上一桶转为 closed，并执行一次最终摘要固化
- 当前新桶以首条消息为起点创建新的 open bucket

## 7. 数据结构方案

## 7.1 表一：`conversation_summary_buckets`

按“会话 + 时间桶”保存桶级摘要。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PK | 本地主键 |
| `session_id` | TEXT | 会话 ID |
| `bucket_start_ts` | INTEGER | 桶起始时间戳 |
| `bucket_end_ts` | INTEGER | 桶结束时间戳；开放桶可等于当前最后一条消息时间 |
| `bucket_rule_version` | INTEGER | 分桶规则版本，默认 `1` |
| `is_open` | INTEGER | 是否开放桶 |
| `anchor_message_id` | TEXT | 当前桶的首条消息 ID |
| `last_message_id` | TEXT | 当前桶最后一条消息 ID |
| `last_message_ts` | INTEGER | 当前桶最后消息时间 |
| `message_count` | INTEGER | 当前桶已纳入摘要的消息数 |
| `summary_status` | TEXT | `pending / queued / processing / ready / stale / failed` |
| `summary_text_ciphertext` | TEXT | 供 prompt 直接读取的压缩文本摘要密文 |
| `summary_json_ciphertext` | TEXT | 结构化摘要 JSON 密文 |
| `summary_version` | INTEGER | 摘要格式版本 |
| `media_item_count` | INTEGER | 当前桶关联的图片/语音摘要数量 |
| `error_code` | TEXT | 最近一次失败码 |
| `notified_at` | INTEGER NULL | 最近一次成功提示时间 |
| `created_at` | INTEGER | 创建时间 |
| `updated_at` | INTEGER | 更新时间 |

推荐唯一约束：

- `UNIQUE(session_id, bucket_start_ts, bucket_rule_version)`

说明：

- `summary_text_ciphertext` 是给推荐回复直接读取的扁平摘要
- `summary_json_ciphertext` 是给未来 UI、调试和再加工预留的结构化摘要
- 两者都建议先经应用侧加密，再写入本地数据库

## 7.2 表二：`conversation_summary_media_cache`

按“消息”缓存图片和语音的摘要结果，避免同一条媒体在每次生成推荐回复时被重复解析。

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PK | 本地主键 |
| `session_id` | TEXT | 会话 ID |
| `message_id` | TEXT | 媒体消息 ID |
| `bucket_start_ts` | INTEGER | 所属时间桶起点 |
| `media_kind` | TEXT | `image / audio` |
| `source_fingerprint` | TEXT | 媒体内容指纹，用于失效和去重 |
| `summary_status` | TEXT | `pending / queued / processing / ready / failed` |
| `summary_text_ciphertext` | TEXT | 供桶摘要/推荐回复读取的短摘要密文 |
| `detail_json_ciphertext` | TEXT | 结构化媒体理解结果密文 |
| `model_name` | TEXT | 生成该摘要的模型名 |
| `runtime_kind` | TEXT | `multimodal_sidecar / local_text_fallback` 等 |
| `attempt_count` | INTEGER | 已尝试次数 |
| `error_code` | TEXT | 最近失败码 |
| `created_at` | INTEGER | 创建时间 |
| `updated_at` | INTEGER | 更新时间 |

推荐唯一约束：

- `UNIQUE(message_id)`

说明：

- 图片和语音摘要先落到这张表
- 桶摘要只聚合已 `ready` 的媒体摘要
- 后续推荐回复不直接读取原始媒体，而是读取媒体摘要缓存

## 7.3 摘要内容结构

建议 `summary_json_ciphertext` 解密后的结构至少包含：

```json
{
  "headline": "这段时间主要在确认周日见面安排",
  "facts": [
    "双方确认了见面城市",
    "对方暂未确定具体时间"
  ],
  "decisions": [
    "周日继续推进"
  ],
  "pending_items": [
    "等待对方确认具体时段"
  ],
  "tone": "friendly",
  "media_refs": [
    "msg-image-1",
    "msg-audio-2"
  ]
}
```

对应的 `summary_text_ciphertext` 则建议压平成短文本，例如：

```text
本段聊天主要在确认周日见面安排；已确认见面城市；具体时间未定；等待对方继续确认。
```

## 8. 加密存储建议

推荐采用两层保护：

1. 字段层加密  
   使用现有应用侧安全存储能力，对 `summary_text_ciphertext`、`summary_json_ciphertext`、`detail_json_ciphertext` 等字段进行加密后再写库。

2. 数据库层加密  
   当本地数据库启用 SQLCipher 时，摘要表天然继承数据库层加密。

落地原则：

- 不把摘要明文写入独立缓存文件
- 不把完整转写、完整 OCR 原文、完整图片路径写入普通日志
- sidecar 若需要临时文件，应在任务完成后立即删除

## 9. 后台任务策略

## 9.1 触发点

摘要任务不应由 UI 控件驱动，而应由消息生命周期驱动。

推荐触发点：

1. 新消息成功入库或进入本地消息缓存后
2. 消息内容被编辑后
3. 图片/语音消息对应的媒体摘要完成后
4. 会话删除时
5. 清空本地聊天状态时

## 9.2 文本消息处理

当新文本消息到达时：

1. 读取该会话当前最后一个 open bucket
2. 用共享时间分桶规则判断是否进入新桶
3. 若仍在同一桶：
   - 更新桶尾消息信息
   - 将 `summary_status` 标为 `stale`
   - 入队一次 debounce 后的滚动摘要刷新
4. 若进入新桶：
   - 将旧桶 `is_open=0`
   - 对旧桶入队一次最终摘要固化任务
   - 新建一个 open bucket
   - 对新桶入队一次初始滚动摘要任务

## 9.3 图片/语音消息处理

当图片或语音消息到达时：

1. 先按时间规则归入当前 open bucket
2. 在 `conversation_summary_media_cache` 中插入 `pending` 记录
3. 后台异步跑媒体摘要
4. 媒体摘要完成后：
   - 更新 `media_cache` 为 `ready`
   - 将所属 bucket 标记为 `stale`
   - 入队一次桶摘要刷新

这样可以保证媒体理解和桶摘要解耦。

## 9.4 debounce 与合并

建议在同一会话、同一 open bucket 上做短时间合并，避免每来一条消息就立即跑一次摘要。

建议初始策略：

- 开放桶滚动摘要：15 到 30 秒 debounce
- 关闭桶最终摘要：立即排队，但优先级高于开放桶滚动刷新
- 媒体摘要完成后的桶刷新：可与开放桶刷新合并

## 10. 图片与语音的模型自身摘要方案

## 10.1 总原则

图片和语音的摘要优先由模型自身完成，而不是先引入独立 OCR/ASR 管线作为默认路径。

但当前主聊天 AI 链路仍是文本 `llama-cpp-python` 调用路径，因此推荐实现形态不是把多模态输入硬塞进现有文本 runtime，而是新增一个独立的本地多模态摘要 runtime / sidecar。

推荐原因：

- 崩溃隔离更好
- 可以单独管理多模态模型加载、取消和超时
- 不污染当前稳定的文本推荐回复链路
- 后续替换模型或 runtime 成本更低

## 10.2 图片摘要输出建议

图片摘要建议由模型输出：

- 一段短摘要
- 可选识别出的关键文本
- 可选结构标签，例如 `screenshot / receipt / chart / chat_capture`

建议进入桶摘要时只保留最小必要信息，例如：

```text
对方发送了一张截图，内容与周日行程安排有关，图片中出现“14:30”时间。
```

## 10.3 语音摘要输出建议

语音摘要建议由模型输出：

- 短转述摘要
- 可选精简转写
- 置信度或质量标记

建议进入桶摘要时只保留短摘要和必要转写片段，例如：

```text
对方发送了一段语音，主要在说明周日晚些时候才有空。
```

说明：

- 音频能力建议单独加 feature flag
- 当本地多模态音频链路不稳定时，可先将状态置为 `failed` 或 `unsupported`
- 不应因为一条语音处理失败而阻塞整个桶摘要

## 11. InfoBar 提示策略

用户要求“后台静默处理，完成后提示”，但提示不能刷屏。

推荐策略：

1. 当前正在查看的会话：
   - 当 open bucket 首次生成 ready 摘要时，提示一次
   - 当 closed bucket 最终摘要生成完成时，提示一次
   - 同一会话 5 分钟内最多提示一次

2. 非当前会话：
   - 不为每个会话单独弹提示
   - 可选做一条合并提示，例如“若干会话的本地摘要已更新”

3. 失败场景：
   - 单次失败只记日志和状态
   - 连续失败达到阈值时，再提示一次 warning

推荐文案方向：

- `本地会话摘要已更新`
- `本地会话摘要已完成`
- `图片/语音摘要更新后，已刷新本段聊天摘要`

## 12. 删除与清理策略

## 12.1 删除单个会话

当 `SessionManager.remove_session(session_id)` 删除本地会话时，应同步：

- 删除 `conversation_summary_buckets` 中该会话记录
- 删除 `conversation_summary_media_cache` 中该会话记录
- 取消该会话所有未完成摘要任务
- 删除 sidecar 可能遗留的临时文件

推荐做法：

- 在数据库层扩展 `delete_session(session_id)`，显式删除摘要表中的关联记录
- 不把清理逻辑散落在 UI 或 controller 层

## 12.2 清空本地聊天状态

当 `clear_chat_state()` 被调用时，应同步清空：

- 所有桶摘要
- 所有媒体摘要缓存
- 所有本地摘要任务状态

## 13. 与推荐回复的衔接方式

本地会话摘要落地后，推荐回复不应再依赖“整周原文直塞”。

推荐读取顺序：

1. 最近 8 到 20 条原始文本消息
2. 当前 open bucket 的滚动摘要
3. 最近 1 到 3 个 closed bucket 的稳定摘要
4. 最近窗口内相关的图片/语音摘要缓存

这样可以把上下文控制在较短 prompt 内，同时保留跨时间段记忆。

## 14. 分阶段落地建议

### 第一期

- 新增摘要表结构
- 接入文本消息的时间分桶和开放桶滚动摘要
- 接入单会话删除和全量清理

### 第二期

- 接入 closed bucket 最终摘要
- 接入 InfoBar 节流与聚合提示
- 推荐回复开始读取桶摘要

### 第三期

- 接入图片多模态摘要 sidecar
- 图片摘要缓存进入桶摘要和推荐回复

### 第四期

- 接入语音多模态摘要 sidecar
- 语音摘要缓存进入桶摘要和推荐回复
- 以 feature flag 控制启用范围

## 15. 当前推荐决策

基于当前项目状态，推荐决策如下：

1. 摘要分桶必须复用现有聊天时间标签规则，而不是绑定 UI 控件创建事件
2. 摘要持久化应采用“字段层加密 + 数据库层加密”双层方案
3. 时间桶必须区分 open bucket 和 closed bucket
4. 图片、语音摘要优先使用模型自身完成，但实现上采用独立本地多模态 sidecar
5. 推荐回复后续只读取本地摘要缓存，不直接全量扫描长历史
6. 视频继续明确排除在本期范围之外
