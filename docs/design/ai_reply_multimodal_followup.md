# AI 推荐回复多模态后续方案

## 1. 文档目标

本文档定义 AssistIM 推荐回复功能的后续扩展范围，重点覆盖以下两类能力：

- 更长历史上下文的接入方式
- 图片、语音内容进入推荐回复上下文的方式

本文档只讨论推荐回复链路，不扩展到 AI 会话、自动发送、群聊自动参与或视频理解。

## 2. 当前基线

当前推荐回复已经具备以下能力：

- 仅在私聊中启用
- 仅处理文本消息
- 基于最近窗口消息构造 prompt
- 输出候选回复，只回填草稿，不自动发送
- E2EE 会话要求本地模型
- 模型未加载时不会由推荐回复链路触发首次加载

当前实现限制：

- 上下文窗口偏短，适合快速回复，不适合跨天跨度的对话承接
- 图片、语音不会进入推荐回复语义上下文
- 视频明确不在后续范围内

## 3. 需求范围

后续要支持的能力：

1. 推荐回复可利用更长历史上下文，但不能简单把一周原始消息全部塞进 prompt
2. 推荐回复后续支持图片和语音
3. 推荐回复后续不支持视频
4. 任何媒体扩展都不能破坏现有 E2EE 本地约束
5. 推荐回复的延迟必须可控，不能因为一条图片或一段语音让每次回复都重新跑全量理解

明确不做：

- 视频内容理解
- 推荐回复自动发送
- 用远端多模态服务直接读取 E2EE 明文
- 每次推荐回复时重新扫描一周全部消息和全部媒体

## 4. 一周历史上下文的推荐方案

### 4.1 不建议方案

不建议把一周内所有原始文本消息直接拼到 prompt 中。原因：

- 当前本地模型上下文有限，原始消息很容易把窗口打满
- 延迟会显著上升
- 最近关键消息会被长历史噪声淹没
- 对推荐回复这种短任务来说，收益通常不成比例

### 4.2 推荐方案

推荐采用三段式上下文：

1. 最近窗口原文
2. 历史摘要
3. 相关历史召回

具体建议：

- 最近窗口原文：保留最近 8 到 20 条消息原文
- 历史摘要：维护一段短摘要，描述过去数天内的主题、结论、待办和情绪基调
- 相关历史召回：根据当前最新消息，从历史中补充 2 到 5 条高相关旧消息

推荐回复真正送给模型的上下文结构应类似：

```text
最近消息：
- 我: ...
- 对方: ...

历史摘要：
- 最近一周主要在讨论 ...
- 已经确认 ...
- 尚未决定 ...

相关旧消息：
- 2026-04-12 对方: ...
- 2026-04-15 我: ...
```

这样可以把“长历史感知”转成“短 prompt 可控输入”，兼顾延迟和效果。

### 4.3 落地顺序

建议按下面顺序落地：

1. 先做最近窗口扩容和摘要槽位
2. 再做一周历史摘要的生成与缓存
3. 最后再做相关历史召回

不建议第一步就上“整周原文直塞”。

## 5. 图片接入推荐回复的推荐方案

### 5.1 需求定义

当最近对话中出现图片时，推荐回复应当能利用图片内容，而不是把这条消息当作空白上下文。

### 5.2 推荐方案

图片不应直接以原始二进制或路径进入推荐回复 prompt，而应先转换为文本摘要缓存。

推荐流程：

1. 收到图片消息
2. 异步生成图片理解结果
3. 写入本地摘要缓存
4. 推荐回复阶段只读取文本摘要

缓存内容建议包含：

- OCR 文本
- 场景简述
- 关键对象
- 是否包含截图、表单、聊天记录、商品、票据等结构化类型标签

推荐进入 prompt 的形式：

```text
对方发送了图片：
- 图片摘要：这是一个聊天截图，主要内容是 ...
- OCR：...
```

### 5.3 为什么这样做

这样做的好处：

- 推荐回复阶段不用重复跑图片理解
- 延迟更稳定
- 更容易做 E2EE 本地约束
- 更方便做摘要质量兜底和缓存失效

## 6. 语音接入推荐回复的推荐方案

### 6.1 需求定义

当最近对话中出现语音消息时，推荐回复应当能利用语音内容，而不是只看到“收到一段语音”。

### 6.2 推荐方案

语音同样不应直接以原始音频进入推荐回复 prompt，而应先做转写和短摘要，再进入缓存。

推荐流程：

1. 收到语音消息
2. 异步执行 ASR 转写
3. 生成一段短摘要
4. 写入本地缓存
5. 推荐回复阶段只读取文本转写或文本摘要

推荐进入 prompt 的形式：

```text
对方发送了语音：
- 转写：...
- 摘要：对方在说明 ...
```

### 6.3 推荐策略

语音转写进入 prompt 时，建议：

- 原始转写保留长度上限
- 超长转写优先压缩成摘要
- 置信度低时标记“转写可能不准确”

这样可以减少 ASR 噪声直接污染推荐回复。

## 7. 为什么不做视频

视频不纳入推荐回复后续范围，原因如下：

- 预处理成本最高
- 需要同时处理关键帧、字幕、ASR、镜头切换
- 对桌面本地链路的时延和资源占用影响最明显
- 在推荐回复场景里，图片和语音已经覆盖更常见且更高价值的输入

后续如果要支持视频，建议单独立项，不并入当前推荐回复演进。

## 8. E2EE 与隐私约束

图片和语音一旦进入推荐回复上下文，必须继续遵守当前 E2EE 规则：

- E2EE 会话中的媒体理解必须优先本地执行
- 不允许把 E2EE 明文图片、音频、转写直接送往未授权远端服务
- 日志不得输出转写全文、OCR 全文或完整媒体路径
- 媒体摘要缓存应只保留推荐回复真正需要的最小文本结果

## 9. 性能建议

推荐方案的核心原则是“预处理异步化，推荐阶段只读缓存”。

性能建议如下：

- 图片理解和语音转写在消息接收后后台异步执行
- 推荐回复生成阶段不做重型媒体处理
- 摘要缓存需要带状态：
  - pending
  - ready
  - failed
- 推荐回复遇到 pending 媒体时，可先退化为文本-only 模式

这样可以避免“用户刚点开会话就被媒体分析阻塞”。

## 10. 推荐落地顺序

建议分三期：

### 第一期

- 先扩展推荐回复 prompt 结构，为历史摘要和媒体摘要预留槽位
- 不改变当前文本-only 行为

### 第二期

- 支持一周历史摘要缓存
- 仍然不接入图片和语音

### 第三期

- 支持图片摘要缓存
- 支持语音转写/摘要缓存
- 推荐回复正式读取图片和语音摘要

## 11. 当前决策

当前阶段先保持：

- 上下文仍以短窗口文本为主
- 推荐回复先改为 4 条
- 风格为 2 条积极推进型 + 2 条保守婉拒型
- 一周历史、图片、语音能力保留到后续迭代

这样可以先把回复风格和交互稳定下来，再逐步扩展上下文和多模态能力。
